# Car Pool Best — технічна документація проєкту

Telegram-бот пошуку попутників (карпулінг, аналог BlaBlaCar) з веб-адмінкою та
Mini App-картою. Документ призначений для розробників, які прийматимуть проєкт
у підтримку.

---

## 1. Стек і загальна архітектура

| Компонент | Технологія |
|---|---|
| Мова | Python 3.13 |
| Бот | aiogram 3.x (long polling) |
| HTTP-сервер | aiohttp (той самий процес, що й бот) |
| БД | PostgreSQL (прод, Railway) / SQLite (локально, файл `rideshare.db`) |
| ORM | SQLAlchemy 2.x (async) |
| FSM-стан бота | Redis (прод) / пам'ять (локально) |
| Планувальник | APScheduler |
| Геокодинг | Google Places API (New) + Geocoding API, fallback OSM |
| Карта Mini App | Leaflet + OpenStreetMap tiles + OSRM (маршрути) |
| Адмінка | aiohttp + Jinja2 + Bootstrap 5 (серверний рендеринг) |
| Хостинг | Railway (деплой з гілки `main` автоматично) |

**Один процес робить усе:** `python main.py` запускає одночасно:
1. Telegram-бота (polling);
2. aiohttp-сервер на порту `$PORT` (за замовч. 8080), який обслуговує:
   - `/webapp/…` — файли Mini App (карта);
   - `/location/{match_id}` — API живого GPS-трекінгу;
   - `/admin/…` — веб-адмінку;
   - `/debug` — діагностична сторінка;
3. APScheduler з фоновими джобами (див. розділ 8).

## 3. База даних (`database/models.py`)

### Таблиці

| Модель | Таблиця | Призначення |
|---|---|---|
| `User` | `users` | Користувач: id = Telegram ID (BIGINT), рейтинг, лічильники поїздок, блокування, домашнє місто |
| `Trip` | `trips` | Заявка на поїздку (і водія, і пасажира — розрізняються полем `role`) |
| `Match` | `matches` | Пара «водій ↔ пасажир» з прапорцями підтвердження та флоу перед виїздом |
| `Rating` | `ratings` | Кожна окрема оцінка 1–5 (хто, кому, за який метч) |
| `RecurringTrip` | `recurring_trips` | Шаблон регулярної поїздки (маршрут + час + маска днів) |
| `SupportTicket` | `support_tickets` | Звернення в підтримку |
| `FAQ` | `faq` | Питання-відповіді (редагуються з адмінки) |
| `BotSetting` | `bot_settings` | Редаговані тексти кнопок/повідомлень (key → value) |
| `DriverLocation` / `PassengerLocation` | | Останні GPS-координати для живого трекінгу |

### Статуси Trip (найважливіше для розуміння логіки)

| Статус | Значення |
|---|---|
| `ACTIVE` | Заявка опублікована, шукаємо попутника |
| `MATCHING` | Є PENDING-метч, чекаємо підтвердження |
| `BOARDING` | Водій узяв пасажира(ів), але ще є вільні місця — заявка видима далі |
| `CONFIRMED` | Поїздку підтверджено обома сторонами (місця вичерпано) |
| `IN_PROGRESS` | Водій натиснув «Виїхав» — заявка зникає з пошуку |
| `CLOSED` | Завершена / скасована / видалена |

### Статуси Match

`PENDING` → (`CONFIRMED` | `REJECTED`) → (`CANCELLED` | `CLOSED`)

- `REJECTED` — відмова ДО підтвердження (явна, таймаут, «перебито іншим метчем»); причина в `rejection_reason`.
- `CANCELLED` — скасування ВЖЕ підтвердженої поїздки; `cancelled_by` + `cancel_reason`.
- `CLOSED` — поїздка відбулась (нормальне завершення).

### Міграції

Alembic НЕ використовується. Схема оновлюється так:
1. Нова **таблиця** → просто додати модель у `models.py` — `Base.metadata.create_all`
   у `init_db()` створить її при старті.
2. Нова **колонка** в існуючій таблиці → додати модель + додати запис у список
   `new_cols` у `database/database.py::_run_migrations()` (ідемпотентний
   `ALTER TABLE ADD COLUMN`). Приклад — колонка `trips.recurring_id`.

## 4. Життєвий цикл поїздки (основний флоу)

```
Створення (driver.py / passenger.py, FSM 5 кроків)
      │  одразу: find_matches_for_trip() + create_match()
      ▼
PENDING-метч → повідомлення обом сторонам «Знайдено підходящий варіант»
      │  handlers/matching.py: «✅ Підтвердити» / «❌ Відмовитися»
      │  (не відповіли — таймаут-джоба скасує, див. розділ 8)
      ▼
CONFIRMED → обидві сторони отримують контакти
      │  за 10 хв до виїзду — нагадування (send_trip_reminders)
      │  водій «🚀 Виїхав» → Trip.IN_PROGRESS, пасажиру приходить карта трекінгу
      │  пасажир «✅ Я на місці»
      ▼
Через 5 хв після часу виїзду (send_rating_prompts):
      Match/Trips → CLOSED, питання «Чи відбулась зустріч?» → оцінка 1–5 ⭐
      + пропозиція «Зробити поїздку регулярною» (якщо ще не регулярна)
```

**Мультипасажирність:** водій із 4 місцями може підтвердити кількох пасажирів.
Поки лишаються місця, його Trip у статусі `BOARDING` і видимий у пошуку.
Логіка місць: `services/matching.py::get_remaining_seats()` (рахує CONFIRMED-метчі).

**Пряма пропозиція:** з розділу «Всі поїздки» користувач може запропонувати
поїздку без власної заявки — створюється тіньова заявка-дзеркало
(`handlers/announcements.py`), ініціатор автопідтверджується, другій стороні
досить одного кліку.

## 5. Регулярні поїздки (`services/recurring.py`, `handlers/recurring.py`)

- Після створення заявки та після завершення поїздки бот пропонує «Зробити
  поїздку регулярною»: Кожен день / Пн–Пт / Обрати дні → час.
- Створюється `RecurringTrip` (шаблон); конкретні Trip лінкуються через
  `Trip.recurring_id`.
- **Механіка:** коли поточний екземпляр закривається (будь-яким шляхом —
  автозакриття, завершення, скасування, видалення), `spawn_next()` створює нову
  ACTIVE-заявку на найближчий підходящий день за маскою. Викликається з УСІХ
  точок закриття — план не «вмирає» від одного скасування.
- `days_mask` — рядок із 7 символів '0'/'1', індекс 0 = понеділок.
- Регулярні заявки **не рахуються в квоту** активних заявок
  (`Trip.recurring_id.is_(None)` у перевірках квоти).
- Зупинити: «Мої поїздки → 🔁 Регулярні → ❌ Скасувати регулярність»
  (`is_active = False`).

## 6. Геокодинг (`services/geo.py`) — найчутливіший модуль

Порядок спроб для текстового вводу адреси:

```
1. Places Autocomplete (New)   ← той самий API, що в застосунку Google Maps;
   + варіант написання 'ьк'→'к'  дає вибір вулиці по кількох містах/селах
2. Places Text Search (New)    ← POI: метро, орієнтири
3. Geocoding API               ← класичний геокодер + власні retry-и
4. OSM (Nominatim + Photon)    ← ВИМКНЕНО за замовчуванням (OSM_FALLBACK_ENABLED)
```

Ключові особливості (не ламати без розуміння):
- **Ключ** — `GOOGLE_MAPS_API_KEY`. У Google Console мають бути ввімкнені:
  *Places API (New)* і *Geocoding API*. Обмеження ключа — по API (не по IP:
  Railway не має статичного IP).
- **Cooldown:** при `REQUEST_DENIED`/квоті API ставиться на паузу 10 хв і
  автоматично повторюється — редеплой не потрібен.
- `_intended_locality()` — розпізнає село/місто як останній токен після коми
  («Ділова 2, Крюківщина»), щоб не підміняти населений пункт.
- `_filter_by_street()` — нечітке порівняння назви вулиці (difflib ≥0.75 +
  входження підрядка): відсіює чужі вулиці, але ловить варіанти написання
  («Подольска» ≈ «Подільська», «Стуса» ⊂ «Василя Стуса»).
- `_city_match()` — нечітке порівняння міст (поріг 0.70): «Киев» ≈ «Київ»,
  «крюковщина» ≈ «Крюківщина».
- Запити з «метро» обробляються Text Search-ем і повертають 1 результат.
- Гео-локація та точка на карті геокодяться реверсом — завжди 1 результат;
  текст — до 7 кандидатів на вибір користувачу.

Якщо Google стане недоступний/дорогий: постав `OSM_FALLBACK_ENABLED=true`
(або прибери `GOOGLE_MAPS_API_KEY` — OSM увімкнеться автоматично).

## 7. Підбір попутників (`services/matching.py`)

- `find_matches_for_trip()` — кандидати протилежної ролі, фільтри:
  1. різні користувачі;
  2. час відправлення ±`MATCH_TIME_DELTA_HOURS` (2 год, `config.py`);
  3. сумісність маршрутів `routes_compatible()` (радіус `MATCH_RADIUS_KM` = 10 км):
     або обидві кінцеві точки поруч, або маршрут одного є під-відрізком іншого
     (пасажиру «по дорозі» водію);
  4. вистачає вільних місць.
- Підбір запускається: при створенні заявки, при прямій пропозиції, і
  **повторно кожні 3 хв** джобою `rematch_active_trips` (щоб заявки, створені
  «не в тому порядку», все одно зметчились). Пара, що вже мала метч (у т.ч.
  відхилений), повторно не метчиться.

## 8. Фонові джоби (`main.py` → APScheduler, усі в `services/notifications.py`)

| Джоба | Інтервал | Що робить |
|---|---|---|
| `auto_close_expired_trips` | 5 хв | ACTIVE/MATCHING із минулим часом → CLOSED (+ spawn регулярних) |
| `send_rating_prompts` | 1 хв | Через 5 хв після виїзду: закриває CONFIRMED-метчі, питає «чи відбулась зустріч», пропонує оцінку і регулярність |
| `send_trip_reminders` | 1 хв | Нагадування за ~10 хв до виїзду (вікно 9–11 хв) обом сторонам |
| `check_pending_match_timeouts` | 1 хв | Непідтверджені метчі: поїздка <1 год — нагадати на 5-й хв, скасувати на 10-й; >1 год — нагадати на 10-й, попередити на 20-й, скасувати на 30-й |
| `rematch_active_trips` | 3 хв | Повторний підбір для ACTIVE-заявок |
| `reload` (bot_settings) | 1 хв | Перечитує редаговані тексти з БД у кеш |

## 9. Веб-адмінка (`admin/`)

- URL: `https://<домен>/admin` . Вхід — по токену `ADMIN_TOKEN` (cookie на 30 днів).
- Адаптивна: на мобільних сайдбар замінюється нижньою панеллю навігації.
- Розділи:
  - **Dashboard** — фільтр періодів (весь час / сьогодні / місяць / минулий
    місяць / рік / довільні дати, AJAX), воронка конверсії, KPI, причини
    скасувань/відхилень, графіки нових юзерів і поїздок;
  - **Поїздки** — список + карта всіх активних, деталка з маршрутом, закриття;
  - **Користувачі** — пошук, блокування, деталка (поїздки, оцінки, тікети),
    ручне редагування рейтингу, надсилання повідомлення від імені бота, CSV;
  - **Метчі** — списки за статусами, деталка з GPS, force-confirm / force-cancel;
  - **Розсилка** — сегментована (всі / водії / пасажири / активні 7-30 дн /
    місто), фонова з live-прогресом; банер у /start;
  - **Матч вручну** — примусове зведення двох заявок;
  - **Безпека** — розподіл оцінок, низькорейтингові користувачі, зриви;
  - **Воркфлоу** — інтерактивна схема бота, клік по кнопці/тексту → інлайн-
    редагування (зберігається в `bot_settings`, бот підхоплює за ≤1 хв);
  - **Звернення**, **FAQ** (CRUD).
- `/admin` УСЕРЕДИНІ бота (`handlers/admin.py`) — скорочена версія: статистика,
  пошук юзера за ID, блокування. Доступ — по `ADMIN_IDS`.

## 10. Mini App (`webapp/`)

Роздається самим ботом за адресою `https://<домен>/webapp/`. Режими
(параметр `?mode=`):
- `single` — маршрут однієї поїздки (з картки);
- `match` — два маршрути: водія і пасажира;
- `track` — живий GPS-трекінг обох сторін (координати шле/читає через
  `/location/{match_id}`, авторизація — Telegram initData HMAC);
- `pick` — вибір точки на карті перетягуванням піна (drag-to-select),
  повертає координати+адресу боту через `sendData`.

## 12. Запуск і деплой

### Локально
```bash
cd bot
pip install -r requirements.txt
copy .env.example .env      # заповнити BOT_TOKEN, ADMIN_IDS, GOOGLE_MAPS_API_KEY…
python main.py
```
Без `DATABASE_URL` створиться SQLite `rideshare.db`; без `REDIS_URL` FSM у пам'яті.

### Railway (прод)
1. Репозиторій підключено до Railway; пуш у `main` → автодеплой
   (`railway.toml`: `startCommand = "python main.py"`).
2. Плагіни: PostgreSQL (обов'язково), Redis (бажано).
3. Всі змінні з таблиці вище — у Variables сервісу.
4. Кастомний домен: Settings → Networking → Custom Domain (+CNAME у DNS).
5. Схема БД оновлюється сама при старті (`init_db()`).

⚠️ Не запускати два екземпляри бота одночасно (у т.ч. локально з прод-токеном) —
Telegram дає `TelegramConflictError`, апдейти губляться.

## 14. Відомі особливості та нюанси

1. **Часова зона.** Всі `departure_time` зберігаються як naive Kyiv time;
   `created_at` — naive UTC. Порівняння «зараз» — тільки через
   `services/timezone.py::now()`. Не використовуйте `datetime.now()` напряму.
2. **Rich Message-картки** (`rich_cards.py`) використовують
   `bot.send_rich_message` (Bot API 10.1); якщо метод недоступний — автоматичний
   fallback на звичайний HTML-текст.
3. **Порядок роутерів** у `main.py` має значення: `fallback.router` мусить бути
   останнім, інакше він перехопить усі повідомлення.
4. **Дані Google по селах неповні** — частина вулиць відсутня або в російському
   написанні. Для таких випадків у боті скрізь є кнопка «🗺 Обрати місце на
   карті» (точка на мапі, без назви вулиці).
5. **`ADMIN_TOKEN`** за замовчуванням «changeme» — обов'язково задати сильний.
6. **Ротація ключів.** Якщо ключ Google або токен бота світилися у листуванні/
   скрінах — перевипустити (Google Console → Regenerate; @BotFather → /revoke).
7. `check_db.py` — допоміжний скрипт перегляду локальної SQLite, у проді не
   використовується.
8. Ліміт історії графіка «за весь час» в адмінці — 36 місяців.
